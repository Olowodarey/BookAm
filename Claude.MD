## What BookAm is

BookAm is a digital **record-keeping** app for Nigerian **ajo / esusu / adashe** coordinators. The name is Nigerian Pidgin for *"record it / write it down."*

It replaces the paper collection card and notebook a coordinator uses to track a rotating savings group. A coordinator creates a **circle**, adds members, and marks contributions as they come in. Everyone sees the same up-to-date view of who has paid, who is owing, and who collects next.

**Target user:** ajo coordinators / collectors ("alajo"), cooperative and market-association treasurers. Phone-first, WhatsApp-centric, trust-driven, not very technical.

### The money rule (critical)
BookAm is **tracking-only**. Members pay each other **directly**, exactly as they do today. BookAm only *records* who paid what. It is **not** a bank, wallet, or payment processor and must never hold or route user funds. This keeps BookAm a software product and out of Nigeria's Mobile Money Operator licensing regime. Any feature that would hold, pool, or transfer contributions is out of scope ‚Äî flag it, don't build it.

## Tech stack

- **Frontend:** `frontend/` ‚Äî Next.js 16 (App Router) + TypeScript + Tailwind CSS v4 (web app + marketing landing + admin console).
- **Backend:** `backend/` ‚Äî NestJS 11 (TypeScript, CommonJS), standalone API server on port **4000**.
- **Database:** PostgreSQL via **Prisma 6** (`backend/prisma/schema.prisma` + migrations).
- **Auth (email-first, Gmail-only):** email is the primary identity ‚Äî one email, one account. **`RegisterDto`, `LoginDto` and `InviteMemberDto` require a `@gmail.com` address** (`@Matches(/@gmail\.com$/i)`) ‚Äî everyone on BookAm uses a Google/Gmail identity; seed accounts use `@gmail.com` too. (Sign-out clears the whole session and lands on the landing page `/`; there are no landing links inside the authed shells.) Everyone registers as a contributor (MEMBER) with **email + password**; the email is verified by an emailed 6-digit code (`EmailOtp` model + `EmailOtpService`; sent over free Gmail SMTP in `backend/src/auth/email.service.ts` ‚Äî set `GMAIL_USER` + `GMAIL_APP_PASSWORD`, else dev builds log the mail and surface the code as `devCode`). "Sign in with Google" (verified server-side via Google's tokeninfo endpoint, no SDK; enabled by `GOOGLE_CLIENT_ID` + `NEXT_PUBLIC_GOOGLE_CLIENT_ID`) signs in directly and **auto-links** to a same-email account ‚Äî Google emails are already verified, so no extra step. A Google-only user adds a password via forgot-password (the emailed code proves ownership); registering a password on an already-verified email is refused (anti-takeover). **Phone/WhatsApp is optional**, verified in-app from the settings pages (authenticated `POST /auth/phone/send-otp` + `/auth/phone/verify`; `PhoneOtp` + `OtpService`, delivered over the WhatsApp Cloud API in `backend/src/auth/whatsapp.service.ts` ‚Äî set `WHATSAPP_PHONE_NUMBER_ID` + `WHATSAPP_ACCESS_TOKEN` + an approved Authentication template `WHATSAPP_TEMPLATE_NAME`, else dev logs the code as `devCode`); verifying a phone auto-claims any circle memberships coordinators created for that number. Becoming a COLLECTOR goes through a `CollectorApplication` (applied from `/me`, approved by the admin). JWT payload is `{ sub, email, role }`; JWT + role guards unchanged. Frontend: unified `/login` + `/register` (email+password, emailed-code step via `components/auth/EmailOtpForm`, Google button), `/forgot-password`, and a "WhatsApp number" verify card in `components/settings/ProfileSettings`. `SafeUser` mirror in `lib/admin/types.ts` (`email` required, `phone` nullable, `emailVerifiedAt`/`phoneVerifiedAt`).
- **Notifications:** WhatsApp / SMS *(e.g. WhatsApp Business API, or a provider like Termii / Africa's Talking)*.
- **Payments:** none held in-app for **ajo contributions** (see "The money rule"). BookAm's **own SaaS subscription fees** (coordinators paying for the software) are legitimate revenue and are managed as records in the admin console; actual charge collection via Paystack is a marked `// TODO: Paystack` integration point in `backend/src/admin/subscriptions.service.ts`.

## Core data model (starting point for the backend)

> These are the domain entities. Refine as you build; keep this section in sync with the actual schema.

Implemented in `backend/prisma/schema.prisma` (‚úì = in the schema today):

- **User** ‚úì ‚Äî a person. May be a coordinator, a member, or both. Identified by **email** (unique, required; `emailVerifiedAt`). Optional `googleId` (linked on same-email Google sign-in), optional `phone` (unique, nullable; `phoneVerifiedAt` ‚Äî verified in-app to claim memberships), and an optional `passwordHash` (Google-only users may have none). Has `role` (MEMBER | COORDINATOR | ADMIN), `status` (ACTIVE | SUSPENDED).
- **CollectorApplication** ‚úì ‚Äî a user's request to become a coordinator: applicant, status (PENDING | APPROVED | REJECTED), applicant's note, reviewNote, reviewedBy, reviewedAt. Approving promotes the applicant to COORDINATOR.
- **SubscriptionPlan** ‚úì ‚Äî a BookAm SaaS pricing tier: name, priceNaira, interval (MONTHLY | QUARTERLY | YEARLY), maxCircles (null = unlimited), features[], active. *This is BookAm's own revenue, not ajo money.*
- **Subscription** ‚úì ‚Äî which user is on which plan: status (ACTIVE | EXPIRED | CANCELLED), periodStart/periodEnd. Records only; Paystack charge collection is a marked TODO.
- **Circle** ‚úì ‚Äî one rotating savings group: name, contribution amount, frequency (daily/weekly/monthly), status (ACTIVE | COMPLETED | PAUSED), coordinator (User), `memberTarget` (planned rotation size), and an optional `inviteToken` for the shareable join link.
- **Membership** ‚úì ‚Äî an **account** in a Circle (`userId` always set; `name`/`phone` copied from their profile). Status: **INVITED** (coordinator added them by email ‚Üí they accept from their dashboard), **REQUESTED** (they asked via the invite link ‚Üí coordinator approves), **ACTIVE** (in the rotation with a 1-based `position`, assigned on activation), **REMOVED** (soft-removed so records survive). Every member is a real account so they can upload their own receipts ‚Äî there is no offline name+phone member anymore. Position uniqueness among active members is enforced in the service layer. **Invite flow:** coordinator `POST /circles/:id/members/invite {email}` ‚Äî **emails the Gmail an invite** (via `EmailService`, exported from AuthModule): if they have an account it links to `/me` to accept; if not, the row is created with `userId: null` + `invitedEmail`, the email links to `/register`, and `AuthService.claimEmailInvites` links it to their account on sign-up/Google sign-in (in `verifyEmail` + `googleSignIn`). Or the shareable link (`GET /invite/:token` public preview; `POST /invite/:token/join` **requires auth**, creates a REQUEST). Coordinator approves/rejects (`.../members/:mid/approve`, `DELETE .../members/:mid/pending`); the member accepts/declines (`/member/invites`, `/member/invites/:mid/accept`, `DELETE /member/invites/:mid`). `CircleDetail` carries `pendingRequests` + `pendingInvites`; the member home shows a "Circle invites" card.
- **Cycle** ‚úì ‚Äî one contribution period/round of a Circle: `index` (unique per circle), status (OPEN | COMPLETED), and the current **collector** (a Membership). One cycle is open at a time; completing a payout opens the next.
- **Contribution** ‚úì ‚Äî one member's payment record for a cycle: amount, status (AWAITING | PENDING_REVIEW | PAID | REJECTED), `receiptFileUrl` (latest proof), rejectionReason, reviewedBy/reviewedAt. Unique per (membership, cycle). **A record only ‚Äî no funds move.**
- **ContributionReceipt** ‚úì ‚Äî installment ledger: a contribution can be paid **bit by bit**, each part a row with `amountNaira` + `receiptFileUrl` + uploader. `Contribution.paidNaira` (API-computed) = sum of receipts; the coordinator still confirms PAID. `Contribution.receiptFileUrl` mirrors the latest. Uploaded by the member (`/member/circles/:id/receipt`) or coordinator (`/circles/:id/contributions/:cid/receipt`), each accepting an optional `amount` form field (defaults to the full amount).
- **Payout** ‚úì ‚Äî the record of the pot handed to the cycle's collector: status (PENDING | COMPLETED), `receiptFileUrl` (latest transfer proof), amount (computed sum of PAID contributions), completedAt. Unique per cycle. **PayoutReceipt** mirrors ContributionReceipt for paying the collector in installments (`/circles/:id/payout/receipt` + optional `amount`). **BookAm does not disburse.**
- **Schedule** ‚úì ‚Äî `Circle.startDate` + `Cycle.dueAt` (both nullable). Create with `startDate` + `firstDueAt` (ISO); each new round auto-advances the deadline by frequency (`advanceDeadline` in `src/circles/schedule.ts` ‚Äî daily +1d, weekly +7d, monthly +1mo) when the next cycle opens in `payouts.complete`. Adjustable via `PATCH /circles/:id {startDate, dueAt}` (dueAt updates the open round). **All deadlines are West Africa Time (fixed UTC+1)** ‚Äî the frontend converts `datetime-local` ‚Üî ISO with `watInputToISO`/`isoToWatInput` and displays via `formatDeadline` (`timeZone: Africa/Lagos`), so every member sees the same time. Shown on the coordinator + member circle pages; editable in the coordinator settings card.
- **Coordinator fee** ‚úì ‚Äî `Circle.coordinatorFeePercent` (0‚Äì100, whole percent; set at create or via `PATCH /circles/:id {feePercent}`). The collector receives **pot ‚àí fee**: `PayoutInfo`/`MemberPayout` carry `feeNaira` + `netPayoutNaira` (computed in `src/circles/fee.ts`), and `MemberCircleDetail.coordinatorFeePercent` is shown to every member. Coordinators can **opt into their own circle's rotation** (`POST`/`DELETE /circles/:id/members/join-self`; `CircleDetail.iAmMember`) ‚Äî running a circle no longer auto-joins them.
- Receipts are **visible to the whole circle** ‚Äî the member circle-detail (`GET /member/circles/:id`) returns every member's `receipts` + the cycle `payout` receipts; the shared `components/circles/ReceiptLedger` renders them on both consoles.
- **Appeal** ‚úì ‚Äî a member's request to be considered to collect next: circle, appellant (Membership), reason, status (OPEN | APPROVED | REJECTED | WITHDRAWN), decidedBy (coordinator User), decidedAt, outcomeNote. One OPEN appeal per member per circle (service-enforced). Reason, tally and outcome are visible to the whole circle.
- **AppealVote** ‚úì ‚Äî one member's advisory vote on an appeal: value (SUPPORT | OPPOSE), unique per (appeal, voter); changeable while the appeal is open. The tally is advisory ‚Äî the coordinator decides, and approval reorders the rotation so the appellant collects right after the current turn.

Key rules encoded: rotation position is managed per-circle in the service layer; completing a payout closes the cycle and advances the rotation (next active member by position who hasn't collected; when everyone has, the circle is COMPLETED); every money-related field is a *label on a record*, never a balance BookAm controls. Receipt files are validated (JPG/PNG/WebP/PDF, ‚â§5 MB) and stored on local disk in dev (`backend/uploads/`, served at `/uploads/*`) behind a storage adapter with a `// TODO: S3/Cloudinary` swap point.

## Domain glossary

- **Ajo / Esusu / Adashe** ‚Äî Nigerian rotating savings & credit associations (ROSCA). Same idea, different regions/languages.
- **Circle** ‚Äî one savings group in BookAm.
- **Coordinator / Collector / Alajo** ‚Äî the person who runs the circle and records payments.
- **Rotation / Payout order** ‚Äî the agreed order in which each member collects the pooled sum.
- **Pot** ‚Äî the pooled contributions collected in one cycle (conceptual; BookAm never holds it).
- **Cycle** ‚Äî one full round of the rotation.
